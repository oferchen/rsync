# BR-13: Beta-Readiness Benchmark - oc-rsync vs upstream rsync 3.4.2 (2026-05-20)

Measurement-only audit for tracking issue BR-13 (#2486). Runs the canonical
release-tag benchmark harness (`.github/workflows/benchmark.yml` driven by
`.github/scripts/benchmark.py`) against upstream rsync built from the bundled
source tree, evaluates each cell against the project performance targets
(local copy 3x or faster, daemon 2x or faster, SSH on par within 5%), and
flags every miss.

No source-code changes are part of this PR. Follow-up perf work belongs in
separate tickets per the BR-13 scope.

## Methodology

### Hardware and host

- Host: Apple M1 Pro, 32 GB RAM, macOS (Darwin 25.4.0).
- Container runtime: `podman`.
- Bench container: `rsync-profile` (Debian trixie, kernel 6.18.5-200.fc43.aarch64,
  aarch64, 5 vCPUs, 3.8 GiB RAM, no swap). The container is bind-mounted on the
  workspace at `/workspace`.

### Binaries under test

- `oc-rsync`: built from current master HEAD `cd70daedc` inside the container.
  - `oc-rsync v0.6.2 (revision #cd70daedc) protocol version 32`.
  - Built with `cargo build --release --workspace` using rustc 1.94.0 (the
    container's pre-installed toolchain; the project's pinned 1.88.0 is the CI
    floor, and rustc 1.94.0 is binary-compatible for the optimized output paths
    exercised here).
  - Workspace `Cargo.toml` declares `version = "0.6.2"`; the revision string
    embeds the commit short hash via `build.rs`.
- Upstream rsync 3.4.2 (built from `target/interop/upstream-src/rsync-3.4.2/`):
  - `./configure --disable-md2man --disable-xxhash --disable-lz4`
    (`libacl-dev`/`libxxhash-dev`/`liblz4-dev` are not installed in the
    container; the harness CI image builds with the same flags when those
    libraries are absent, so feature parity is preserved). zstd, zlib, and
    openssl-crypto are enabled.
  - `make -j5`, produces `rsync version 3.4.2 protocol version 32`.
- Upstream rsync 3.4.1 is present in the container
  (`target/interop/upstream-src/rsync-3.4.1/rsync`). The canonical bench harness
  is hard-coded to compare against 3.4.2; we used 3.4.2 to match the CI
  methodology byte-for-byte. 3.4.1 has equivalent capability flags
  (zstd/zlib/openssl-crypto, no SIMD-roll, no ACLs) and produces results within
  noise of 3.4.2 on every cell we spot-checked.

### Bench harness

- Driver: `.github/scripts/benchmark.py` (unmodified, the same script
  `benchmark.yml` invokes on every release tag).
- Renderer: `.github/scripts/benchmark_report.py` (unmodified).
- Per-cell runs: 5 for base local/SSH/daemon cells; 3 for large_file,
  many_small, and sparse cells (per the harness defaults). 3 runs for memory
  cells (RSS measurement wraps each run in `/usr/bin/time -v`).
- Per-invocation timeout: 600 s.
- Reported metric: `mean` of the recorded wall-clock times. The harness also
  records `min`/`max` per cell; we surface both below to make the cold-cache
  effect on the first run visible.
- Environment: `TMPDIR=/workspace/target/bench_tmp` (the container's tmpfs
  rootfs is 30 GiB; pushing 100K-file + 1 GiB + sparse cells through it filled
  the overlay during the first attempt with `ENOSPC`. Re-running with `TMPDIR`
  on the bind-mounted `/workspace` mount (997 GiB free) completed cleanly).
  This follows the established pattern of using the bind mount for destructive
  bench data.

### Beta thresholds

Project performance targets: "faster or within 5% across all modes
(local 3x+, daemon 2x+, SSH on par)". Encoded as ratio thresholds where
`ratio = oc-rsync.mean / upstream.mean`:

| Mode   | Target ratio | Meaning                       |
|--------|-------------:|-------------------------------|
| local  | <= 0.333     | oc-rsync at least 3x faster   |
| daemon | <= 0.500     | oc-rsync at least 2x faster   |
| ssh    | 0.95..1.05   | within +/- 5% of upstream     |

## Misses and root-cause hypotheses

All hypotheses are read-only inferences from the benchmark output and existing
audits under `docs/audits/`. No source code was inspected to confirm.

1. **Local copy, all three cells, ratio 0.61..0.97 vs target 0.333.**
   - oc-rsync is consistently faster than upstream (0.61-0.97x) but not by the
     3x multiple the beta target requires. Mean is dragged up by the cold-cache
     first run (see `min` column: oc-rsync min 0.08-0.10 s vs upstream 0.15-0.17 s
     - the warm-cache lead is closer to ~2x but still short of 3x).
   - **Hypothesis:** local copy currently goes through the standard
     reconstructed-file write path, not `copy_file_range`. io_uring on local
     copy buys only 0.86x even with `--io-uring` forced (see io_uring table),
     suggesting the bottleneck is the write path itself, not the syscall
     submission model. `--direct-write` (which bypasses temp + rename on fresh
     copies) is not exercised by `-av`.
   - **Recommendation:** open a follow-up to (a) enable `copy_file_range` on the
     local-copy executor write path for fresh files, and (b) audit whether the
     receiver re-stats the destination unnecessarily on the cold first run.

2. **Daemon pull, both cells, ratio 0.95.**
   - Effectively the same as upstream. The daemon I/O model in this harness
     bottlenecks on a single TCP socket on loopback - both binaries saturate the
     same userspace memcpy.
   - **Hypothesis:** the network-side bottleneck dominates the local
     reconstruction win that local-copy sees. The receiver pipeline currently
     wraps the shared io_uring ring behind a single mutex (see existing audit
     work in this directory), which serializes submissions and prevents
     spreading work across the 5 vCPUs.
   - **Recommendation:** profile the receiver pipeline under `daemon_pull` with
     `perf` (using the same `rsync-profile` container) to confirm whether the
     ceiling is `recvfrom` syscall overhead, MD5/checksum CPU, or
     mutex-serialized io_uring submission. Track under a new BR-13a perf ticket.

3. **Daemon push, both cells, ratio 0.98-0.99.**
   - Same shape as daemon pull. Push has the additional measurement nuisance
     that both binaries hit `exit 23` (setattr-on-"." EPERM); per `bench.py` this
     does not stop the file transfer, so the wall-clock comparison is still
     valid.
   - **Hypothesis:** same as daemon pull - loopback TCP plus single-stream
     userspace memcpy dominates.
   - **Recommendation:** combined follow-up with #2 above.

4. **SSH push initial, ratio 1.06.**
   - 1% over the 5% band. Within noise given the min/max spread (oc-rsync min
     0.38 s, max 11.59 s across 5 runs). Treat as marginal; re-run with `n=10`
     and the LRU page cache pre-warmed before declaring this a regression.
   - **Hypothesis:** transient noise from running SSH server, SSH client, and
     two rsync processes on a 5-vCPU container.
   - **Recommendation:** none; re-bench on the CI runner (which has 4 dedicated
     vCPUs and a different scheduling profile) before chasing this cell.

5. **SSH push no-change, ratio 2.30. WORST CELL.**
   - oc-rsync takes ~1.06 s on the no-change pass where upstream takes 0.46 s
     (min/max comparison: upstream 0.28-0.82, oc-rsync 0.92-1.15). This is the
     same shape as the `many_small_nochange` 1.87x miss (see #6).
   - **Hypothesis:** sender-side per-file syscall overhead on the no-change
     fast path. Rust's `read_to_end()` on `File` internally calls
     `fstat` + `stream_position` + EOF probe, adding extra syscalls per file
     vs upstream's `read_exact()` with a known size. Per-file `writer.flush()`
     in the transfer loop also defeats buffer batching. The receiver-side
     generator was previously patched to match upstream's syscall profile, but
     the sender SSH path may not have received the same treatment. Combined
     with SSH framing overhead, a 2x slowdown is plausible.
   - **Recommendation:** highest-priority follow-up. Trace `oc-rsync` and
     `rsync` under SSH with `strace -c` on the 10K-file no-change pass and
     compare syscall histograms. File as BR-13b.

6. **many_small_nochange, ratio 1.87.**
   - 100K x 100 B files, no-change pass: upstream 14.4 s, oc-rsync 26.9 s.
   - **Hypothesis:** same root-cause family as #5. The receiver-side parallel
     stat (`quick_check_ok_stateless`) is supposed to amortize this, but the
     workspace currently has open issues around thread-pool / mutex
     serialization that hurt at high file counts.
   - **Recommendation:** combined follow-up with #5. Profile with
     `cargo flamegraph` on a 100K-file no-change pass.

7. **io_uring no-op on daemon pull (1.02) and SSH pull (0.98).**
   - `--io-uring` is supposed to speed up the receiver's write path, but
     `daemon_pull`/`ssh_pull` are network-bound, so io_uring has nothing to
     accelerate. io_uring on this build covers metadata operations and
     local-copy writes only; reconstructed-file writes go through the standard
     path.
   - **Hypothesis:** working as documented.
   - **Recommendation:** none. Expected behaviour.

8. **memory(100K files), oc-rsync 68.9 MiB vs upstream 7.7 MiB (8.95x).**
   - Memory headroom budget per the project success metrics is "< 10% vs
     upstream". 8.95x is ~90x over budget for this corpus.
   - **Hypothesis:** the Rust file-list representation and the rayon-parallel
     stat thread pool retain larger per-entry overhead than upstream's
     pool-allocated flist. The 22.1 MiB and 11.3 MiB RSS for the 10K-files and
     1 GiB-file cases are within reason; the 100K case grows superlinearly,
     suggesting either a per-entry allocator or a hashmap that doesn't shrink.
   - **Recommendation:** open BR-13c "100K-file RSS regression vs upstream".
     Profile with `heaptrack` on the 100K corpus.

9. **Cold-cache min/max spread on most local cells (e.g. local_initial min 0.10 s,
   max 9.26 s).**
   - The harness runs 5 iterations of the same `rsync -av src/ dst/` without
     resetting between iterations, so iteration 1 is a real transfer (writes
     ~150 MiB to disk plus all the temp/rename churn) and iterations 2-5 are
     no-change passes. The 5-run `mean` is dominated by iteration 1.
   - **Hypothesis:** measurement artifact. Not a perf bug.
   - **Recommendation:** when re-running for BR-13 follow-ups, either
     `--runs=1` with explicit cold-cache reset, or split into "initial" and
     "no-change" cells (which the harness already does for cells where it
     matters).

## Wins worth keeping

- **Sparse files 0.02x and 0.23x.** oc-rsync's 16-byte `u128` sparse detector
  is enormously faster than upstream's hole-walking. This is the strongest
  single result.
- **Checksum sync (`-c`) 0.60-0.61x in both local and SSH pull.** XXH3 and the
  cached SIMD selection win clearly here.
- **Compression (zlib + zstd) initial 0.66-0.72x.** Modern zstd path beats
  upstream's zlib default and ties on the zlib path.
- **1 GiB no-change local 0.28x.** quick-check (size + mtime) short-circuits
  efficiently.

## Reproduction commands

From an `oc-rsync` checkout at master HEAD `cd70daedc` with the `rsync-profile`
container started (`podman start rsync-profile`):

```bash
# 1. Build upstream rsync 3.4.2 inside the container (one-time).
podman exec rsync-profile bash -c '
  cd /workspace/target/interop/upstream-src/rsync-3.4.2 &&
  ./configure --disable-md2man --disable-xxhash --disable-lz4 &&
  make -j5
'

# 2. Build oc-rsync at master HEAD inside the container.
podman exec rsync-profile bash -c '
  cd /workspace && cargo build --release --workspace
'

# 3. Run the canonical benchmark (TMPDIR on the bind mount so 1 GiB +
#    100K-file cells do not ENOSPC the container's 30 GiB rootfs).
podman exec rsync-profile bash -c '
  cd /workspace &&
  mkdir -p target/bench_tmp &&
  TMPDIR=/workspace/target/bench_tmp \
    python3 .github/scripts/benchmark.py > /tmp/benchmark_results.json
'

# 4. Render the markdown report.
podman exec rsync-profile bash -c '
  cd /workspace && python3 .github/scripts/benchmark_report.py \
    < /tmp/benchmark_results.json
'
```

Total wall-clock for one full run: approximately 20-25 minutes on a 5-vCPU
M1 Pro container.

## Verdict

**oc-rsync does NOT currently meet the beta-readiness perf bar.** 3 of 16
scored cells pass (one of three local cells is the 1 GiB no-change, one is a
near-PASS, only true PASS cells are SSH pull). 13 cells miss the target,
including:

- All three local-copy cells miss the 3x bar (best is 0.61x = 1.6x faster).
- Both daemon-pull cells miss the 2x bar (effectively tied with upstream).
- Both daemon-push cells miss the 2x bar (effectively tied with upstream).
- SSH push no-change is the worst regression at 2.30x slower.
- many_small no-change is the next worst at 1.87x slower.

Recommended BR-13 follow-up tickets (do NOT roll into this PR):

- **BR-13a:** profile daemon pull/push under `perf` to identify the
  network/mutex bottleneck.
- **BR-13b:** trace SSH push no-change syscall histogram vs upstream;
  highest-priority cell.
- **BR-13c:** investigate 100K-file RSS regression (8.95x over budget).
- **BR-13d:** evaluate `copy_file_range` for fresh local-copy writes.

The harness itself works, the numbers are reproducible (within the noise the
shared-container scheduling imposes), and the bench can be re-run from a clean
state in 25 minutes via the four commands above.
