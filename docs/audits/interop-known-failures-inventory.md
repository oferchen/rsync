## Top 5 remediation targets

Ranked by cost-vs-impact. Cost is the engineering surface that has to move;
impact is "tests turned from XFAIL to PASS" plus "interop scenarios unblocked
end-to-end."

### 1. `daemon` / `daemon-gzip-download` / `daemon-gzip-upload`

- **Entries:** `tools/ci/upstream_testsuite_known_failures.conf:28-30`
- **Root cause:** Upstream's `daemon.test` builds an `rsyncd.conf` via
  `build_rsyncd_conf` in `target/interop/upstream-src/rsync-3.4.1/testsuite/rsync.fns`
  with `list = no`, multiple modules (`test-from`, `test-to`, `test-scratch`,
  `test-hidden`), and a custom `log format` directive. The harness then
  invokes `oc-rsync --daemon --config=$conf` via the test fixture. The
  daemon-config parser at
  `crates/daemon/src/daemon/sections/config_parsing/module_directives.rs`
  must accept every directive upstream emits, with semantics matching
  upstream's `loadparm.c`. Three tests at once depend on this single
  surface; the gzip variants additionally need `daemon-gzip-download` /
  `daemon-gzip-upload` plumbing through the daemon-mode compression path
  exercised in `tools/ci/run_interop.sh:8316-8340` (delta-stats fixture
  builds the same daemon shape).
- **Cost:** Medium. Two failure modes: (a) directives parsed but
  effect-not-applied, (b) directives the parser rejects. Both are
  surveyable by running `WHICHTESTS=daemon.test tools/ci/run_upstream_testsuite.sh`
  once locally and reading the resulting `target/interop/upstream-testsuite/daemon.log`.
- **Impact:** 3 testsuite entries clear in one PR; daemon-config interop
  becomes provably upstream-shape.
- **Fix shape:** Walk every directive upstream emits in `build_rsyncd_conf`,
  diff against `ModuleBuilder::set_*` setters in
  `crates/daemon/src/daemon/sections/module_definition/setters.rs`, and add
  the missing apply-time wiring (mirroring how `charset` was wired in
  `crates/daemon/src/daemon/sections/module_access/client_args.rs:283`).
  No protocol work, no wire changes.

### 2. `hardlinks`

- **Entry:** `tools/ci/upstream_testsuite_known_failures.conf:51`
- **Root cause:** `hardlinks.test` triggers INC_RECURSE sender-side paths
  that are explicitly disabled today. The capability string is built at
  `crates/transfer/src/setup/capability.rs::build_capability_string` and
  passes `!is_sender` for the `'i'` (INC_RECURSE) flag, so push transfers
  do not advertise inc-recurse even when the receiver supports it.
  Documented in `docs/audits/compatibility-flags-transfer-matrix.md:40,49`.
- **Cost:** Medium-to-large. Sender-side INC_RECURSE state machine exists in
  the generator but has never been validated against upstream; switching
  it on is "the work."
- **Impact:** 1 testsuite entry; broader: removes the last functional
  asymmetry in the sender vs receiver protocol path.
- **Fix shape:** Wire sender-side INC_RECURSE flag emission, verify
  segmented file-list dispatch matches upstream `flist.c:send_directory()`
  + `generator.c:recv_files()`, then flip
  `build_capability_string(!is_sender)` -> `build_capability_string(true)`.

### 3. `atimes`

- **Entry:** `tools/ci/upstream_testsuite_known_failures.conf:56`
- **Root cause:** `--atimes` propagation to receiver. Upstream wire-encodes
  atimes alongside mtime at protocol 30+ when `--atimes` is in effect.
- **Cost:** Small-to-medium. Single option, single wire field, single
  receiver apply path. Touchpoints are surveyable from
  `target/interop/upstream-src/rsync-3.4.1/flist.c:721-754` (send) and
  `flist.c:1080-1110` (recv) plus the corresponding oc-rsync `read_extras`
  / `write_extras` files.
- **Impact:** 1 testsuite entry, plus removes a `--atimes` gap visible from
  the CLI.
- **Fix shape:** Mirror upstream's flag-gated atime field. Already partially
  wired; remaining work is generator -> receiver application during the
  metadata phase.

### 4. `crtimes`

- **Entry:** `tools/ci/upstream_testsuite_known_failures.conf:57`
- **Root cause:** Birth-time on the receiver. The metadata crate's birth
  time path is platform-dependent (macOS `birthtime`, Linux `statx`,
  Windows `GetFileInformationByHandle`).
- **Cost:** Small per-platform; coordination cost across the three
  platforms is the real overhead.
- **Impact:** 1 testsuite entry. Closes a long-standing platform-coverage
  gap.
- **Fix shape:** Implement `--crtimes` receiver apply step using the
  existing `metadata` crate creation-time helpers.

### 5. `chmod-temp-dir`

- **Entry:** `tools/ci/upstream_testsuite_known_failures.conf:62`
- **Root cause:** `--temp-dir` + `--chmod` interaction. The upstream test
  pins the tempfile on a different filesystem than the destination, then
  validates that chmod modes (including setuid/setgid/sticky bits)
  survive the cross-fs rename.
- **Cost:** Small. The local-copy executor at
  `crates/engine/src/local_copy/executor/file/` already supports
  cross-filesystem temp-dir renames; the gap is likely permission-mode
  application order.
- **Impact:** 1 testsuite entry.
- **Fix shape:** Audit the order in which chmod and rename happen on the
  cross-fs path; ensure setuid/setgid/sticky survive the final move.

## Recommended next-fix target

**`tools/ci/upstream_testsuite_known_failures.conf:28-30` - the
`daemon` / `daemon-gzip-download` / `daemon-gzip-upload` cluster.**

Justification:

- **Highest test-clearance ratio per PR.** One audit-and-wire pass against
  upstream's `build_rsyncd_conf` directive set clears three of the 17
  testsuite entries in a single change.
- **Smallest scope per entry cleared.** The work is entirely inside
  `crates/daemon/src/daemon/sections/config_parsing/` and
  `crates/daemon/src/daemon/sections/module_definition/setters.rs`. No
  protocol changes, no platform-specific code, no metadata crate work.
  Compare to `hardlinks` (INC_RECURSE sender) or `crtimes` (three
  platforms), which need wire-format or platform coordination.
- **High interop value.** Daemon-config parity is a project-wide goal
  (oc-rsync is the binary mode of `oc-rsyncd`, per `CLAUDE.md`
  Architecture). Clearing these three tests proves daemon-config interop
  against upstream's canonical config shape, not just our own fixtures.
- **No upstream-version risk.** Unlike the protocol-locked entries
  (`up:acls`, `up:xattrs`, `up:compress-zstd`, `up:compress-lz4`,
  `up:merge-filter`), every directive upstream emits is supported on
  every protocol version we negotiate.
- **Companion work already exists.** `tools/ci/run_interop.sh:8316-8340`
  already builds a daemon fixture for delta-stats. The same harness
  scaffolding can be reused to dump upstream's actual rsyncd.conf during
  a `daemon.test` failure for a one-shot diff.

Implementation plan in one PR:

1. Run `WHICHTESTS=daemon.test tools/ci/run_upstream_testsuite.sh` once;
   archive `target/interop/upstream-testsuite/daemon.log` plus the
   generated `rsyncd.conf` for diff against
   `crates/daemon/src/daemon/sections/config_parsing/module_directives.rs`.
2. Add any missing directive parsers and `ModuleBuilder` setters; wire
   them through `client_args.rs` the same way `charset` was wired at
   line 283.
3. Repeat for `daemon-gzip-download.test` and `daemon-gzip-upload.test`
   (gzip variants exercise the same config + `compress` directive plus
   download/upload module flags).
4. In the same PR, delete the three entries from
   `tools/ci/upstream_testsuite_known_failures.conf:28-30` and update
   this inventory's category table.

Acceptance: `WHICHTESTS='daemon*.test' tools/ci/run_upstream_testsuite.sh`
prints `PASS daemon`, `PASS daemon-gzip-download`, `PASS daemon-gzip-upload`
and no `UPASS` row.
