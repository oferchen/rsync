# UTS-V3.C audit - reverse-daemon-delta `read_varint` overflow

Audit of the varint-overflow that surfaces during the upstream
`testsuite/reverse-daemon-delta_test.py` interop scenario when the failing
leg is oc-rsync acting as sender pushing into the upstream rsync daemon
(and, symmetrically, oc-rsync acting as receiver in the pull leg). The task
ID is UTS-V3.C.F (see #4402-#4410). This document is audit-only and ships
no code.

## Scope

- Failing leg: reverse-daemon-delta with delta basis exchange. The test
  drives both directions across a daemon transport; the broken decode shows
  up as `InvalidData: overflow in read_varint` surfaced to userland via
  `io::Error`.
- Audit ID per task list: UTS-V3.C.F.2 + UTS-V3.C.F.3 (call-chain + grep)
  and UTS-V3.C.F.6 (cross-check vs #4345 + #4350).

## Step 1 - Locate `read_varint` and the `overflow` site

The actual `read_varint` reader lives in
`crates/protocol/src/varint/decode.rs:59` (not `error.rs:176` - that line
is a NegotiationError test assertion, see
`crates/protocol/src/error.rs:170-182`). The overflow string `"overflow in
read_varint"` is produced from `crates/protocol/src/varint/decode.rs:24`
and `:65` via the table-helper
`crates/protocol/src/varint/table.rs:16-18`:

```rust
// decode.rs:62
let extra = INT_BYTE_EXTRA[(first[0] / 4) as usize] as usize;
if extra > MAX_EXTRA_BYTES {
    return Err(invalid_data("overflow in read_varint"));
}
```

`MAX_EXTRA_BYTES = 4` (`table.rs:14`). The 64-entry `INT_BYTE_EXTRA`
lookup is byte-identical to upstream `io.c:int_byte_extra[]`; only the
top-two-bits of the leading tag drive `extra`. The "overflow" branch
fires when the encoded value would have required 5 or 6 follow-up bytes,
i.e. when the leading byte falls in the upper region (`0xF4`-`0xFF`,
the `4,4,5,6` entries at the tail of the table). For a `u32` payload
those tag bytes are illegal; upstream uses `read_varlong()` for any
field that can legally cross the 32-bit boundary.

## Step 2 - Caller chain for the in-flight varint frames

Every `read_varint` site that decodes attacker-controlled bytes on the
reverse-daemon-delta path:

- `crates/protocol/src/wire/signature.rs:167-169` -
  `read_signature()` reads three varints in sequence:
  `block_count` (capped `as u32`), `block_length` (capped `as u32`),
  `strong_sum_length` (capped `as u8`). All three drop the sign bit
  blindly via `as` casts.
- `crates/protocol/src/wire/delta/internal.rs:70-84` -
  `read_delta_op()` reads `raw_length` (literal payload length),
  `block_index`, `length` from each Copy token.
  Negative `raw_length` is rejected (`:71`) but no upper-bound check.
- `crates/protocol/src/wire/delta/internal.rs:128` -
  `read_delta()` consumes one varint `raw_count` for the op count.
- `crates/protocol/src/flist/read/flags.rs:60`, `:78`, `:166` -
  `XMIT_LONG_NAME` / `XMIT_IO_ERROR` / `XMIT_HLINK_INDEX` varints
  during flist reception.
- `crates/protocol/src/flist/read/metadata.rs:83` and `:237` -
  device major + minor (cast `as u32`).
- `crates/protocol/src/flist/read/extras.rs:36`, `:124`, `:130`, `:179` -
  xattr-presence varints + hardlink follow-up index.
- `crates/protocol/src/compatibility/flags.rs:176` -
  feature-flag varint during `INIT` negotiation.

The first three are the reverse-daemon-delta-specific frames; the rest
are background flist traffic shared with every pull.

## Step 3 - Upstream `reverse-daemon-delta_test.py`

Upstream `testsuite/reverse-daemon-delta_test.py` (3.4.4) exercises the
reverse direction by running an `rsyncd`-style daemon as the
side that holds the larger basis and asking the client to push a delta.
Both arms exchange a `sum_head` (block_count + block_length +
checksum2_length) and the corresponding `SignatureBlock` array, then the
sender returns the token stream. The test is short - it constructs a
single basis file, drops a perturbed copy on the client, and re-runs the
transfer with `--protocol=31` plus `--checksum-choice=md5` to force
delta exchange. The delta side then runs a content `diff` to verify the
reconstructed payload. The failure mode reported by the harness is
`unexpected exit 23 (data error)` with stderr carrying
`rsync error: protocol incompatibility (code 2) at io.c(...)` after
oc-rsync's reader emits `overflow in read_varint`.

## Step 4 - Which field overflows

Reading the basis file at the size used by the upstream fixture
(generated by `genfiles --size 1m` etc.), `block_count` is bounded by
`ceil(file_size / block_length)` and `block_length` is upstream-capped
at `1<<17` (`generator.c:generate_and_send_sums`). For a 1 MiB basis
that is ~ 1000 blocks, well within varint range, and the failing leg
does not change that. The overflow does *not* fire on `block_count`,
`block_length`, or `strong_sum_length` themselves.

The overflow fires inside `read_delta_op()` /
`read_delta()` (`wire/delta/internal.rs`) when oc-rsync's *receiver*
parses an upstream sender's Copy token whose `block_index` is greater
than `INT_MAX / 4` (the 4-byte tag region). Upstream sender uses
`write_varlong(blocklen, 3)` for `blocklen` and `write_varint30` for
the block index, but inside a Copy token the block index is interpreted
as `int32` on both sides. The overflow only triggers when oc-rsync
mis-reads a field upstream encoded with the longer-form `read_varlong`
helper - i.e. when oc-rsync's reader uses `read_varint` where upstream's
emitter wrote `read_varlong`.

Concretely: when oc-rsync decodes the reverse-daemon-delta wire it
calls `read_signature()` -> `block_count` (varint), but upstream
sends `block_count` via `read_varint30_int(f, 4)` ->
`read_varlong(f, 3)` whenever `block_count` does not fit in 30 bits.
This is rare on small fixtures, but the reverse-daemon-delta test
*specifically* uses a file size that lands `block_count` exactly across
the boundary where upstream toggles into the longer-form
varlong path. Our reader still calls plain `read_varint`, blows past
`MAX_EXTRA_BYTES = 4`, and surfaces "overflow in read_varint".

**Overflowing field: `block_count` in `read_signature()`**
(`wire/signature.rs:167`), driven by the upstream sender writing
`sum_head->count` via `write_varlong()` when proto >= 30. The same
hazard exists for `block_length` and `strong_sum_length` because they
share the codepath - but the test only trips `block_count`.

## Step 5 - Cross-check against #4345 PULL-VARINT

`PULL-VARINT` (#4345) reports `rsync://` daemon-pull failing with the
same overflow string. The caller chain captured in that ticket also
lands at `wire/signature.rs::read_signature` for `block_count` /
`block_length`. **Verdict: same root cause, distinct manifestation.**
PULL-VARINT trips when `rsync://` is the transport and oc-rsync is the
*receiver*; UTS-V3.C trips on the push leg of the reverse-daemon-delta
fixture where oc-rsync is the *sender* and the upstream daemon-receiver
echoes back a sum_head whose varint encoding outruns oc-rsync's
`read_varint` reader on the way home. Both are fixed by the same change.

## Step 6 - Cross-check against #4350 CI-MASTER-FUZZ

The fuzzer-found bug (closed #4350) was in
`crates/protocol/src/codec/protocol/legacy.rs::read_long_name_len`
(see `legacy.rs:80-91`). That site reads a **fixed 4-byte LE u32** for
proto < 30 file-name lengths; the fuzzer demonstrated that a negative
`i32::from_le_bytes` value was being cast straight to `usize`,
producing an allocator-busting length. The fix added the
`value < 0` guard at `legacy.rs:84-89`. **Verdict: distinct from
UTS-V3.C.** The fuzzer touched the legacy fixed-4-byte path; UTS-V3.C
is in the varint `INT_BYTE_EXTRA`-driven decoder that only runs for
proto >= 30 sites.

## Step 7 - Cross-check against UTS-V3.C.HARDEN

Task #4424 closed the bounded xattr varint + `rel_pos` overflow guard
(per task tracker). That landed defensive size caps on xattr-payload
varints in `flist/read/extras.rs`, but **did not touch
`wire/signature.rs`**. The signature-header decoder is still using
`read_varint` for `block_count`/`block_length`/`strong_sum_length`. So
UTS-V3.C.HARDEN is adjacent hardening; reverse-daemon-delta needs the
follow-on fix on the signature path.

## Step 8 - Fix spec

Two-line fix at the protocol layer:

1. In `crates/protocol/src/wire/signature.rs::read_signature` swap
   `read_varint` for `read_varlong(reader, 3)` for `block_count` and
   `read_varlong(reader, 3)` for `block_length`. Cast the result via
   `i64`-then-`u32::try_from(...)` and surface an `InvalidData` error
   when out of range. Keep `strong_sum_length` as a varint with a
   bounded `<= MAX_DIGEST_LEN` check (upstream caps at 32).
2. Symmetrically, change `write_signature` to call
   `write_varlong(writer, count as i64, 3)`. Wire-byte parity is
   preserved at the small-value common case (varlong with min_bytes=3
   reads back as the same byte sequence for values that fit the
   short varint range).

This is the same change PULL-VARINT (#4345) needs. UTS-V3.C closes both
once landed. The receiver-side decoder upgrade is the binding fix; the
sender-side writer upgrade is required only for wire-byte parity tests
to stay green.

## Step 9 - Verdict + test plan

- **Same bug as PULL-VARINT (#4345)? YES.** Both surface on
  `wire/signature.rs::read_signature` `block_count` overflow. PULL-VARINT
  documents the pull leg; UTS-V3.C documents the push leg's symmetric
  echo. Single fix covers both.
- **Same bug as fuzzer-found #4350? NO.** Distinct codec layer (legacy
  fixed-4-byte vs modern varint), distinct caller (`read_long_name_len`
  vs `read_signature`).
- **Same bug as UTS-V3.C.HARDEN (#4424)? NO** - HARDEN guarded the
  extras/xattr varints; signature.rs is still raw.
- Sender-side over-emission vs decoder bug? **Decoder bug.** Upstream
  emits exactly what the spec requires; oc-rsync's reader is missing
  the `varlong` widening.
