# Wire format differential fuzzer for protocol bytes

Design note for the differential layer that sits on top of the existing
bytes-only cargo-fuzz roundtrip harness in `crates/protocol/fuzz/`. The
existing harness (#1194) feeds libFuzzer arbitrary structured inputs and
asserts encode/decode roundtrip. This note specifies the next layer:
take a *known-good* protocol byte stream captured from upstream rsync
3.4.1, mutate one byte (or a small region), replay through oc-rsync's
decoder, and either confirm rejection with a structured error or
confirm acceptance with output equivalent to upstream's output on the
same mutated input.

This is a design note. No Rust code lands in this PR. Follow-up work
items are listed in section 13 and tracked outside the persistent TODO
list. Implementation is gated behind a new `wire_diff_fuzz` cargo-fuzz
target that extends the workspace-disjoint harness already configured
in `crates/protocol/fuzz/Cargo.toml`.

## 1. Problem statement

Bytes-only roundtrip fuzz (#1194) excels at exercising the encoder and
decoder symmetrically on inputs that the encoder accepted. It does not
explore the asymmetric attack surface that real protocol traffic
presents: a peer (or man-in-the-middle) who sends bytes that *look*
like protocol traffic, satisfy length checks, parse past the first few
fields, and then encode something subtly malformed in a deeper layer.

Random byte streams generated by libFuzzer's mutation engine almost
never look like real rsync traffic. They reach shallow rejection paths
quickly because the very first byte is wrong, or the magic prefix is
wrong, or the multiplex frame header decodes to an oversized payload
that the decoder rejects before parsing the body. Coverage stalls in
the first 100 bytes of the parser.

Real-world protocol divergence comes from *plausible* byte streams:
streams that pass the entry checks, get most of the way through a
file-list entry or delta token, then encode a malformed length, an
out-of-range varint, a name with embedded NUL, a hardlink index that
points past the end of the list. Mutation of a known-good capture is a
higher-yield strategy because the mutation lands in a region the
decoder is already committed to parsing.

The proposal: collect a corpus of pre-recorded upstream rsync byte
streams, apply a small set of mutation strategies, and replay through
oc-rsync's decoder under libFuzzer's coverage instrumentation. Compare
the reaction to upstream's reaction on the same mutated stream. Any
divergence is a fuzz finding.

## 2. Scope

The fuzzer covers every wire frame and stream that oc-rsync exposes
through the `protocol` crate's public decode API. The list below
mirrors the `pub mod` and `pub use` declarations in
`crates/protocol/src/wire/mod.rs`:

- **Variable-length integers**: `read_varint`, `read_varlong`,
  `read_varlong30`, `read_longint`, `read_int` from
  `crates/protocol/src/varint/mod.rs` and `crates/protocol/src/varint/decode.rs`.
- **Delta tokens**: literal-data, block-match, end-marker, and
  whole-file paths from `crates/protocol/src/wire/delta/mod.rs`,
  `delta/token.rs`, `delta/internal.rs`, `delta/types.rs`,
  `delta/int_encoding.rs`.
- **File entries**: every encoder/decoder pair re-exported from
  `crates/protocol/src/wire/file_entry/mod.rs` and
  `crates/protocol/src/wire/file_entry_decode/mod.rs`. The decoder
  tree spans `checksum.rs`, `device.rs`, `flags.rs`, `hardlink.rs`,
  `mode.rs`, `name.rs`, `ownership.rs`, `size.rs`, `symlink.rs`, and
  `timestamps.rs`.
- **Signature blocks**: `read_signature` /
  `crates/protocol/src/wire/signature.rs`.
- **NDX values**: file-list and special index frames embedded in the
  delta and goodbye phases.
- **Multiplex frames**: `MessageHeader`, `MessageFrame`,
  `BorrowedMessageFrame`, `recv_msg`, `recv_msg_into` reachable
  through `crates/protocol/src/multiplex/io/mod.rs`,
  `multiplex/io/recv.rs`, `multiplex/io/send.rs`, and the underlying
  `multiplex/frame.rs` and `multiplex/codec.rs`.
- **Compressed token streams**: zlib, zstd, and lz4 paths in
  `crates/protocol/src/wire/compressed_token/mod.rs`,
  `compressed_token/decoder.rs`, `compressed_token/zlib_codec.rs`
  (#1114), `compressed_token/zstd_codec.rs` (#1115), and
  `compressed_token/lz4_codec.rs` (#1116). The per-token flush path is
  what permits a single corrupt byte in a compressed stream to derail
  the next token boundary; that is exactly the area the fuzzer
  targets.
- **Filter wire**: `crates/protocol/src/filters/wire.rs` for the rare
  paths that send rule bytes inline.
- **File list**: `crates/protocol/src/flist/wire_mode.rs` and
  `flist/wire_path.rs` for path-encoding edge cases.
- **xattr / ACL**: indirectly reachable through file-entry frames when
  `--xattrs` or `--acls` is negotiated.

All five protocol versions (28, 29, 30, 31, 32) are in scope. The
version-aware decoders use the strongly-typed `ProtocolVersion` enum
from `crates/protocol/src/version/protocol_version/mod.rs` (per #1250)
for branch selection. The fuzzer enumerates all five versions per
capture rather than picking one, because version-specific bugs (such
as the v28/v29 4-byte LE vs varint-30 boundary) only surface on one
version while the surrounding bytes look identical.

## 3. Capture corpus

The corpus is a tree of pre-recorded upstream rsync sessions stored as
raw byte streams (one per direction, post-demultiplex where
applicable). The on-disk layout is:

```
tests/wire-corpus/
  v28/
    push/
      basic.bin
      acl.bin
      xattr.bin
      hardlinks.bin
      sparse.bin
      compress-zlib.bin
      compress-zstd.bin
    pull/...
    local/...
  v29/...
  v30/...
  v31/...
  v32/...
  manifest.toml
```

The naming convention is `{protocol}-{direction}-{feature}-{compression}.bin`,
so a fully qualified capture is `tests/wire-corpus/v32/push/acl-xattr-zstd.bin`.
The flat name is recorded in the per-target manifest for fast lookup.

`manifest.toml` records, for each capture file:

- the upstream rsync source build identity (commit hash and tarball
  digest of `rsync-3.4.1.tar.gz`)
- the negotiated protocol version
- the direction (push / pull / local)
- the negotiated capability flags
- the cleartext command line that produced it
- the SHA-256 of the capture
- the total wire byte count and the number of multiplex frames
- the host kernel and rsync build flags

Captures are generated by the existing interop harness driver
(`tools/ci/run_interop.sh`) plus a new `tools/ci/capture_wire.sh`
helper that runs upstream rsync against a fixed test fixture, tees the
TCP stream to a file, and records the manifest entry. Captures are
trimmed to the protocol body (greeting and TLS framing stripped) so
the fuzzer feeds the decoder the same bytes the multiplex layer would
hand it after the handshake completes.

The corpus is checked into the repository at
`tests/wire-corpus/`. Captures are small (most are <64 KiB; the
largest sparse-file fixture is ~512 KiB) and the version-pinning
manifest makes regeneration a pure CI script invocation.

A separate `tests/wire-corpus/seeds/` directory holds minimal hand-
crafted seeds that exercise individual frame types in isolation:
single-token deltas, single-entry file lists, single-frame multiplex
exchanges. These seeds give libFuzzer something to mutate even when
the full session captures are unavailable.

## 4. Mutation strategies

The mutation engine is deterministic given a `(capture, seed,
strategy)` tuple. Each strategy picks an offset uniformly within the
capture (or within a discoverable structural region; see
length-field-corrupt below) and applies a single edit. Mutations
compose: libFuzzer can apply two mutations to the same input by
running the engine twice.

The six strategies, in order of expected yield:

1. **Single-byte flip**: pick offset `i`, replace `bytes[i]` with a
   random byte. Lowest mechanical cost, hits parser branches
   uniformly. Catches missing range checks.

2. **Byte insert**: pick offset `i`, insert one random byte; all
   following bytes shift right by one. Total length increases by one.
   Catches off-by-one in length-prefixed regions.

3. **Byte delete**: pick offset `i`, remove `bytes[i]`; all following
   bytes shift left by one. Total length decreases by one. Catches
   short-read handling and EOF detection.

4. **Length-field corrupt**: parse the stream forward (using a
   lightweight structural walker, not the full decoder) until the
   first varint that names a length. Increase or decrease that varint
   by one without re-encoding the trailing payload. The decoder will
   read the length, expect N bytes, and either find too few or too
   many. This is the highest-yield strategy because it directly
   exercises the trust boundary between length fields and payload.

5. **Truncate**: drop everything from offset `i` to the end. Verifies
   that every decode path handles short reads with `UnexpectedEof`,
   not panic.

6. **Repeat**: pick an offset and length, duplicate that region in
   place. Stresses streaming decoders that maintain per-frame state
   (the multiplex layer, the compressed-token decoder).

Each strategy has a per-iteration probability that defaults to uniform
(1/6) but the manifest can override per-capture weights. For example,
captures with a single multiplex frame benefit more from
length-field-corrupt than from repeat.

A seventh, **compressed-stream-aware** mutation is reserved for
captures whose body is a compressed token stream. Mutating a single
byte of zlib/zstd/lz4 output usually invalidates the whole stream
trivially. The aware mutation decompresses, mutates the inner bytes,
re-compresses, and re-frames. Without this, the per-token flush points
introduced for `flush=Z_SYNC_FLUSH` resync (#1114, #1115, #1116) get
no real exercise from the byte-flip strategy. Implementation reuses
`CompressedTokenDecoder` and `CompressedTokenEncoder` from
`compressed_token/decoder.rs` and `compressed_token/encoder.rs`.

## 5. Equivalence oracle

The oracle decides whether a given (capture, mutation, decoder
reaction) tuple is a finding. The decision tree:

1. **Crash on either side**: panic, abort, segfault, or any
   uncaught error. Always a finding. cargo-fuzz already detects
   panics on the oc-rsync side; the upstream-diff layer adds the
   same check on the upstream side via a wrapper that runs upstream
   rsync under a SIGABRT-detecting harness.

2. **Both reject**: oc-rsync returns `Err(_)`; upstream rsync exits
   non-zero. Confirm structural equivalence:
   - exit codes match (both 12, both 14, etc.; both must map to the
     same upstream-compatible code per the role-trailer contract,
     #996-#999)
   - stderr role trailer (`[sender]`, `[receiver]`, `[generator]`,
     `[server]`, `[client]`, `[daemon]`) matches
   - error category (protocol vs I/O vs auth) matches
   - no panic, no UB, no leaked file descriptors

   Exact stderr text is not required to match (oc-rsync's diagnostics
   are intentionally clearer in some places). The structural fields
   listed above are the contract.

3. **Both accept**: oc-rsync produces a complete decode tree;
   upstream's receiver writes the destination file. Confirm semantic
   equivalence:
   - same set of files written, same content, same metadata
     (mode, owner, group, mtime, xattrs, ACLs)
   - same exit code (typically 0)

   Byte-identical *output bytes* are not the contract: a mutation
   that changes the order of two unrelated file-list entries may
   produce the same destination tree via a different wire path. The
   contract is destination tree equivalence, not byte-trace
   equivalence.

4. **One rejects, one accepts**: divergence. Always a finding. This
   is the most valuable signal because it identifies a region where
   the two implementations disagree about whether bytes are valid.

The oracle runs in two modes:

- **Fast mode** (per-PR CI): only oc-rsync is invoked. Crashes on the
  oc-rsync side and structural sanity checks on the result. No
  upstream comparison. ~10K iterations per PR target.

- **Diff mode** (nightly batch): both implementations are invoked on
  the same mutated stream. Full equivalence check. ~1M iterations per
  nightly run.

Fast mode is in-process. Diff mode forks an upstream rsync subprocess
per iteration; this is slow but acceptable for an overnight run.

## 6. Implementation site

The new fuzz target lives next to the existing ones in
`crates/protocol/fuzz/fuzz_targets/wire_diff_fuzz.rs`. It is added to
`crates/protocol/fuzz/Cargo.toml` as a sibling `[[bin]]` entry. The
fuzz crate already has the `[workspace]` directive that detaches it
from the root workspace, so the addition does not affect any other
crate's build graph.

Sketch of the target shape (illustrative; not the implementation):

```text
#![no_main]

use libfuzzer_sys::fuzz_target;

fuzz_target!(|data: &[u8]| {
    // 1. Pick a capture from the embedded corpus by hashing `data[..2]`.
    // 2. Use the rest of `data` as a mutation seed.
    // 3. Apply one strategy (selected by data[2..4]).
    // 4. Replay through oc-rsync's decoder.
    // 5. Compare to the precomputed upstream reaction (loaded from
    //    the manifest in fast mode; recomputed in diff mode).
});
```

The manifest is loaded once at process start via `OnceLock`. The
corpus files are bundled into the target binary via `include_bytes!`
so the fuzzer is self-contained and reproducible without filesystem
state.

The mutation engine is its own module (`mutate.rs`) with unit tests
in `crates/protocol/fuzz/src/mutate_tests.rs`. Mutation is a pure
function over `(capture: &[u8], seed: &[u8], strategy: u8) -> Vec<u8>`,
which means the same mutation is reproducible from a libFuzzer crash
artifact.

## 7. Wire-compat invariants

The fuzzer is test-only infrastructure. It MUST NOT change any of the
following:

- Production decoder behaviour. The fuzzer calls existing public
  decode APIs; it does not introduce new acceptance criteria, new
  error variants, or new validation paths. If a fuzz finding requires
  a code change, that change lands in a separate PR and is reviewed
  on its own merits.
- Wire byte format. The fuzzer reads bytes; it never instructs an
  encoder to write bytes that upstream would reject.
- Protocol version negotiation. Captures pre-record the negotiated
  version; the fuzzer drives the decoder with the recorded version.
- Multiplex framing rules. The same `MessageHeader::decode` and
  `recv_msg` from `crates/protocol/src/multiplex/io/recv.rs` are
  invoked; no test-only fork.
- Capability flags. The fuzzer does not toggle capability bits at
  decode time; the recorded capabilities are honored.

The only addition to production code is a feature-gated `pub fn` in
the `protocol` crate that exposes the structural walker (used by the
length-field-corrupt strategy). That walker is opt-in via a
`coverage_counters` cargo feature and is not compiled into release
builds.

## 8. Resource budget

CI budget is the binding constraint. Per-PR fast mode:

- 10000 libFuzzer iterations
- target wallclock: ~30 seconds on a CI runner
- in-process, single-threaded, no upstream fork
- runs in the existing fuzz CI lane alongside the other targets

Nightly diff mode:

- 1000000 libFuzzer iterations
- target wallclock: ~6 hours on a nightly batch runner
- forks upstream rsync per iteration (cached in a tmpfs to amortize
  startup)
- runs in a dedicated workflow that does not block PR merge

Coverage from the fast mode is collected via `cargo llvm-cov`
(integrated per #1107) and uploaded as a CI artifact. The coverage
report is the primary signal that the corpus is reaching the right
parts of the decoder.

Memory budget is 256 MiB per iteration in fast mode, 1 GiB in diff
mode (the upstream subprocess is the large consumer). libFuzzer's
`-rss_limit_mb` is set accordingly.

A timeout of 5 seconds per iteration in fast mode and 60 seconds in
diff mode catches infinite loops without false positives on the
slower deeply-nested decode paths.

## 9. Failure handling

When the oracle reports a finding:

1. libFuzzer writes the failing input to `crash-<sha>` in the
   target's artifact directory.
2. cargo-fuzz `tmin` is invoked automatically (in CI) to shrink the
   crash to a minimal failing input.
3. The shrunk input is committed to
   `crates/protocol/tests/wire_fuzz_regressions/<sha>.bin`. A small
   manifest entry records the mutation strategy, the original capture,
   the mutation offset, and the expected oracle decision.
4. A regression test is added to
   `crates/protocol/tests/wire_fuzz_regressions.rs` that loads each
   `.bin` and asserts the documented oracle decision. This test runs
   in the standard nextest matrix on every PR thereafter, so once a
   bug is fixed it stays fixed.
5. The original full capture and the mutation point are dumped
   alongside the regression test so future debuggers can reconstruct
   the exact byte trail.

Diff-mode findings additionally record both stderr/stdout outputs
(oc-rsync's and upstream's) so the structural-equivalence comparison
can be re-run without re-invoking either binary.

## 10. Coverage measurement

Coverage drives the corpus. Two layers:

- **Branch coverage**: the standard `cargo llvm-cov` integration
  (#1107) runs across the fast-mode fuzz target and reports per-line
  hit counts. The CI threshold for the wire decoder is >95% line
  coverage; the fuzzer is expected to keep the wire-decode subtree
  above this threshold even as new frame types are added.

- **libFuzzer feedback**: the `coverage_counters` cargo feature
  enables `#[cfg(fuzzing)]` instrumentation in the wire decoder,
  emitting a counter per branch into libFuzzer's edge map. libFuzzer
  uses this to bias mutation toward unexplored branches. The same
  counters are read by `cargo llvm-cov` in fast mode for the per-PR
  coverage report.

The `coverage_counters` feature is gated behind `cfg(fuzzing)` so it
never appears in release builds. The instrumentation is pure
counter-increment with no observable behaviour change; the fuzz CI
lane verifies this by running the wire-decode unit tests with the
feature enabled and comparing test results against the no-feature
build.

## 11. Risks and mitigations

**Capture corpus drift**. Upstream rsync evolves; the v3.4.1 captures
will eventually mismatch a future v3.5.x format. Mitigation: every
capture records the upstream commit hash and tarball digest in the
manifest, and CI rejects captures whose digest does not match a known
upstream release. Re-capture on each upstream release is a one-shot
script (`tools/ci/recapture_wire.sh`).

**Mutation generates valid-but-different input**. A single-byte flip
can change a uid by one, which the decoder accepts but produces a
different destination metadata. The oracle handles this: it accepts
"both implementations agree on the mutated semantics", not
"destination is byte-identical to the original". The destination tree
equivalence check uses metadata fingerprints (sha256 of file content,
exact mode/uid/gid/mtime), so divergent-but-consistent outcomes are
green, divergent-and-inconsistent outcomes are findings.

**Compressed-stream mutation collapses to no-op**. Flipping one byte
inside zlib's deflate stream usually breaks the stream's CRC and
fails decompression on both sides identically; the oracle returns
green and the iteration was wasted. Mitigation: the
compressed-stream-aware mutation strategy decompresses, mutates the
inner bytes, re-compresses, and re-frames. This is what gives the
zlib/zstd/lz4 per-token flush paths real test coverage.

**Capture leaks operator data**. Captures are derived from synthetic
fixtures only (the existing interop fixture tree under
`target/interop/`); no production traffic is recorded. The CI script
that produces captures asserts the source path is under
`target/interop/` before recording.

**Fuzzer flakiness**. libFuzzer's mutation is non-deterministic in
multi-thread mode; this fuzzer runs single-threaded
(`-jobs=1 -workers=1`) so the same `(capture, seed, strategy)` tuple
always produces the same input. Crashes are reproducible from the
artifact alone.

**Slow upstream subprocess**. Forking upstream rsync per iteration
costs ~50 ms of process-startup overhead. Mitigation: the upstream
binary is run from a tmpfs, the test fixture is also on tmpfs, and
the diff-mode budget assumes 50 ms per iteration. Nightly only.

**False findings from diagnostics divergence**. oc-rsync's error
messages are intentionally clearer than upstream's in some places.
The oracle compares structural fields (exit code, role trailer,
error category), not free-form text. A pure diagnostic-text
divergence is not a finding.

## 12. Interaction with the behavioural fuzzer (#1193)

The behavioural protocol fuzzer (#1193) and this wire-byte
differential fuzzer are complementary, not redundant:

- **#1193 (behavioural)**: generates random end-to-end scenarios at
  the API surface (start a transfer, kill the connection, restart,
  re-negotiate, send a delta, ack, ...). The fuzzer drives the state
  machine, not the wire bytes. It catches state-machine bugs and
  ordering hazards.

- **#1303 (this fuzzer)**: takes a known-good wire byte stream and
  mutates one byte. The fuzzer drives the decoder, not the state
  machine. It catches parser bugs and length-field hazards.

Both run in the same harness crate (`crates/protocol/fuzz/`), share
the corpus directory, and report findings into the same regression
test suite (`crates/protocol/tests/wire_fuzz_regressions.rs`). The
two targets are scheduled as siblings: fast modes run on every PR,
diff modes run nightly.

A finding in one fuzzer often points at the other. A
behavioural-fuzzer crash on a length mismatch leads to a wire-fuzzer
regression covering the exact byte sequence. A wire-fuzzer crash on a
state-dependent decoder branch (e.g., a multiplex frame received in
the wrong session phase) leads to a behavioural-fuzzer regression
covering the state transition that produced it.
