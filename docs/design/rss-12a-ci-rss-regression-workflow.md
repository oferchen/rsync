# RSS-12.a: CI workflow for RSS regression detection

Task: RSS-12.a. Branch: `docs/rss-12a-ci-workflow-spec`. Parent series:
RSS-8..11 (arena-allocated paths migration). Downstream: RSS-12.b
(promotion to required check after bake-in).

## 1. Motivation

The RSS-8..11 series migrates `FileEntry` path storage from per-entry
`PathBuf` + `Arc<Path>` to arena-backed `Spur` handles, reducing peak
RSS from 3-11x upstream to a target of 1.5-2x. Once that migration
lands, we need a CI gate to catch regressions before they reach master.

Without an automated regression gate, per-entry overhead can creep back
through innocent changes - adding a field, boxing something
unnecessarily, or introducing a secondary allocation per entry. The
existing `scripts/benchmark_flist_memory.sh` script provides the
measurement methodology but is manual-only. This spec defines the
GitHub Actions workflow that automates that measurement on every PR
touching memory-sensitive paths.

## 2. Design principles

1. **Absolute threshold, not relative comparison.** Comparing against
   upstream rsync requires building upstream from source on every run,
   adding 3-5 minutes of CI time. Instead, store an absolute peak-RSS
   ceiling (in MB) as a checked-in baseline that tightens over time.
2. **Deterministic fixture.** A fixed synthetic fixture eliminates
   variance from filesystem state, network, or random data.
3. **Single platform.** RSS measurement on Linux via `/usr/bin/time -v`
   (MaxRSS field) is well-understood and stable. macOS `getrusage` uses
   different units (bytes vs KB) and Darwin page-caching makes RSS
   non-deterministic. Linux-only for the gate; macOS as informational.
4. **Low-cost.** The workflow must complete in under 10 minutes on
   `ubuntu-latest` to remain practical as a PR check.
5. **Non-blocking initially.** Ships as `continue-on-error: true` for a
   bake-in period on master. RSS-12.b promotes it to a required check.

## 3. Fixture design

### 3.1 Scale selection: 100K files

A 100K-file fixture strikes the balance between signal and speed:

| Scale | Build time | Transfer time | RSS signal | CI budget fit |
|-------|-----------|--------------|-----------|---------------|
| 10K   | < 1 s     | < 1 s        | Too low - noise dominates | Yes |
| 100K  | ~3 s      | ~5 s         | Strong - per-entry overhead visible at 88 B * 100K = 8.8 MB inline | Yes |
| 1M    | ~30 s     | ~60 s        | Best     | No (> 10 min with build) |

At 100K entries, a 4 B per-entry regression is 400 KB - measurable
above RSS noise (typically +/- 200 KB between runs). A 16 B per-entry
regression (e.g., re-adding `PathBuf`) is 1.6 MB - well above noise.

### 3.2 Fixture structure

```
bench-rss/src/
  dir-{0000..0099}/           # 100 directories
    file-{0000..0999}.dat     # 1000 files per directory (100K total)
```

Each file contains a fixed 64-byte payload (deterministic, no random
data). Directory depth is intentionally shallow to keep path lengths
uniform (~30 chars) and avoid triggering any deep-nesting code paths
that would conflate RSS measurement with recursion overhead.

The fixture is generated by an inline shell script within the workflow
(no committed fixture files - they would bloat the repo).

### 3.3 Transfer mode

**Receiver-side file list build (daemon pull, `--no-inc-recursive`).**

This mode forces the receiver to buffer the entire 100K-entry file list
in memory before transfer begins, exercising the `Vec<FileEntry>`
allocation path that RSS-8..11 optimize. The `--no-inc-recursive` flag
disables INC_RECURSE streaming, ensuring the full flist is held
simultaneously.

Command:
```bash
oc-rsync -r --no-inc-recursive rsync://127.0.0.1:$PORT/bench/ $DST/
```

## 4. Measurement methodology

### 4.1 Tool: `/usr/bin/time -v`

GNU time's "Maximum resident set size (kbytes)" field reports the
high-water mark of the process's RSS from the kernel's perspective.
This is the same metric `scripts/benchmark_flist_memory.sh` uses.

```bash
/usr/bin/time -v oc-rsync ... 2>&1 | grep "Maximum resident"
```

The value is extracted, converted to MB (integer division by 1024), and
compared against the baseline.

### 4.2 Sampling

Run the transfer **3 times** and take the **median** peak RSS. Three
runs is sufficient because:
- The fixture is deterministic (no random I/O patterns).
- The flist build path is allocation-dominated, not I/O-dominated.
- Observed variance in `scripts/benchmark_flist_memory.sh` at 100K is
  < 1% between runs.

### 4.3 Warm-up

One warm-up run (discarded) before the 3 measured runs. This populates
the filesystem cache so that I/O latency does not inflate RSS through
page-cache attribution on the first run.

## 5. Baseline storage strategy

### 5.1 Checked-in JSON baseline

A single file at `.github/baselines/rss-100k.json`:

```json
{
  "fixture": "100k-flat",
  "peak_rss_mb": 42,
  "threshold_percent": 10,
  "updated": "2026-06-01",
  "notes": "Post RSS-8..11 arena migration baseline"
}
```

**Why checked-in, not artifact-based:**
- Artifact comparison requires fetching the prior run's artifact, which
  fails on the first run of a new branch or after artifact expiry (90
  days).
- A checked-in baseline is version-controlled: we can see exactly when
  and why it changed.
- Baseline updates are explicit PRs with review, preventing silent
  ratcheting.

### 5.2 Updating the baseline

When legitimate changes reduce RSS (e.g., further optimizations land),
the baseline is updated by:
1. Running the workflow manually (`workflow_dispatch`).
2. Verifying the new median across 5+ runs.
3. Submitting a PR that updates `peak_rss_mb` in the JSON file.

The threshold is never increased without explicit review and justification.

## 6. Threshold and failure criteria

```
PASS:  median_rss_mb <= baseline.peak_rss_mb * (1 + threshold_percent / 100)
FAIL:  median_rss_mb >  baseline.peak_rss_mb * (1 + threshold_percent / 100)
```

With `peak_rss_mb: 42` and `threshold_percent: 10`:
- Ceiling = 42 * 1.10 = 46.2 MB
- Any median above 46 MB (integer) fails the check.

**Why 10%:**
- At 100K entries, 10% of 42 MB = 4.2 MB = ~42 bytes per entry.
  This catches a single new `u32` field added to `FileEntry` (4 B *
  100K = 400 KB) only if several such regressions accumulate - avoiding
  false positives from single-field additions while catching structural
  regressions like re-introducing `PathBuf` (24 B * 100K = 2.4 MB).
- Tighter than 10% would cause false positives from allocator jitter
  and kernel RSS accounting noise.
- Looser than 15% would miss meaningful regressions.

## 7. Workflow specification

### 7.1 Triggers

```yaml
on:
  workflow_dispatch:
  schedule:
    - cron: '47 2 * * *'   # Nightly at 02:47 UTC
  pull_request:
    paths:
      - 'crates/protocol/src/flist/**'
      - 'crates/protocol/src/recv/**'
      - 'crates/core/src/session/**'
      - 'crates/engine/src/**'
      - '.github/workflows/bench-rss.yml'
      - '.github/baselines/rss-100k.json'
```

**Path filtering rationale:**
- `flist/` - file list construction, entry types, interner.
- `recv/` - receiver-side flist decode and storage.
- `session/` - orchestration that wires flist into transfer.
- `engine/` - file entry consumers that might clone/retain entries.

### 7.2 Concurrency

```yaml
concurrency:
  group: bench-rss-${{ github.ref }}
  cancel-in-progress: true
```

### 7.3 Job structure

Single job: `bench-rss` on `ubuntu-latest`, timeout 20 minutes.

Steps:
1. **Checkout** - standard `actions/checkout`.
2. **Install Rust toolchain** - `dtolnay/rust-toolchain` pinned to 1.88.0.
3. **Cache cargo builds** - `Swatinem/rust-cache` with key `bench-rss`.
4. **Build oc-rsync (release)** - `cargo build --release -p oc-rsync`.
5. **Generate fixture** - shell loop creating 100K files.
6. **Write daemon config** - rsyncd.conf for the `[bench]` module.
7. **Start daemon** - `oc-rsync --daemon --no-detach`.
8. **Warm-up run** - single transfer, output discarded.
9. **Measure RSS (3 runs)** - `/usr/bin/time -v` capturing MaxRSS.
10. **Compute median and assert** - Python one-liner, compare to baseline.
11. **Generate step summary** - markdown table in `$GITHUB_STEP_SUMMARY`.
12. **Stop daemon** - kill PID (always runs).
13. **Upload results** - artifact with raw measurements JSON.

### 7.4 Platform coverage

| Platform | Role | Rationale |
|----------|------|-----------|
| Linux (`ubuntu-latest`) | Gate (required after bake-in) | Stable `/usr/bin/time -v`, deterministic RSS |
| macOS | Not included | `getrusage` reports RSS in bytes (not KB), page-cache inflation, high CI cost. Add as informational-only in a later iteration if needed |
| Windows | Not included | No equivalent of MaxRSS tracking in standard tooling; WSL overhead distorts measurements |

### 7.5 Status: non-required

Ships with `continue-on-error: true`. The workflow posts results to the
step summary and uploads artifacts. Promotion to a required branch
protection check is tracked as RSS-12.b, gated on:
- 2 weeks of stable nightly runs without false positives.
- Post-RSS-8..11 baseline established from real measurements.

## 8. Daemon configuration

The benchmark uses a local daemon (not SSH) to isolate the receiver's
flist memory from SSH process overhead:

```ini
use chroot = no
port = $PORT
pid file = $WORK/rsyncd.pid
log file = $WORK/rsyncd.log

[bench]
    path = $WORK/src
    read only = yes
    list = yes
```

Port is dynamically allocated (Python `socket.bind(0)` trick, same
pattern as `bench-daemon-coldstart.yml`).

## 10. Future extensions

1. **RSS-12.b: promotion to required check.** After bake-in period,
   remove `continue-on-error` and add to branch protection.
2. **1M-scale nightly.** A separate nightly-only job at 1M files
   (longer runtime acceptable) to catch super-linear scaling
   regressions that 100K might miss.
3. **INC_RECURSE mode comparison.** Once sender-side INC_RECURSE
   (#1862) is stable, add a second measurement with default
   (streaming) flist to verify that peak RSS stays bounded.
4. **Baseline auto-ratchet.** A workflow that, on master merges,
   checks if RSS decreased and opens a PR to lower the baseline -
   keeping the gate tight automatically.
5. **Per-crate RSS attribution.** Use `heaptrack` or `jemalloc`
   profiling to attribute RSS to specific crates, enabling targeted
   optimization.

## 11. Relationship to existing harnesses

| Harness | Scope | Automation |
|---------|-------|-----------|
| `scripts/benchmark_flist_memory.sh` | Manual, all scales, all modes | None (container) |
| `crates/protocol/benches/file_entry_memory.rs` | Criterion bench, inline size | `cargo bench` |
| **This workflow** | CI gate, 100K, daemon pull | GitHub Actions |

This workflow complements the manual script by automating the
regression-critical path. It does not replace the manual script, which
remains the tool for exploratory measurement at 1M scale and for
profiling inside the container.

## 12. Risks and mitigations

| Risk | Mitigation |
|------|-----------|
| GHA runner RSS variance between hardware generations | 10% threshold absorbs typical 2-3% variance; nightly runs detect drift early |
| Fixture generation time inflating timeout | 100K files in shell loop takes ~3 s; parallelize with `seq \| xargs -P` if needed |
| Baseline goes stale after optimization | Nightly runs will show consistent PASS with headroom; auto-ratchet (future) closes the gap |
| False positive from unrelated memory (page cache, mmap) | `--no-inc-recursive` forces heap-dominant path; daemon mode avoids SSH overhead; warm-up primes page cache |
| Kernel RSS accounting quirks (THP, mmap) | Release build with default allocator (no jemalloc) on standard GHA Ubuntu; consistent across runs |
